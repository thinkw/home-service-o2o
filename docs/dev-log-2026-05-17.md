### 三、踩坑记录

#### 1. Java 11 兼容性
- `instanceof` 模式匹配（`val instanceof Integer i`）是 Java 16+ 特性，改为传统写法
- MyBatis-Plus 无 `insertOrUpdate()` 方法，改为 `insert` / `updateById` 分支

#### 2. Axios POST + params 陷阱
- 请求 wrapper 对 POST 无 `data` 时会把 `params` 移入请求体
- `@RequestParam` 找不到参数报 `MissingServletRequestParameterException`
- 修复：前端改用 GET，后端合并 GET/POST 逻辑

#### 3. 响应双重序列化
- Spring 返回 `String` 时，`PackResultFilter` 会双重序列化
- 前端 `res.data` 拿到的是 JSON 字符串而非对象
- 修复：前端 `JSON.parse` 解析内层 JSON

#### 4. 前端超时
- AI 总结生成需 ~20 秒，前端默认 10 秒超时
- 修复：该 API 单独设置 180 秒超时

#### 5. JSON 时间戳类型
- Hutool 序列化 `LocalDateTime` 为 Long 时间戳，非字符串
- `(String) lastEval.get("createTime")` 抛 `ClassCastException`
- 修复：按 Number/String 分支处理，区分秒/毫秒级时间戳

| `ai-chat/index.vue` | 前端 | 接线 `@stop` → `handleStop`，路由离开时取消请求 |

#### 取消链路
```
用户点击停止 → reader.cancel() + AbortController.abort() + DELETE /cancel
  → Java: SSE 断开 → cancelSession → 发送 cancel 帧 → 关闭 WebSocket
  → Python: cancel_event.set() + tool_ctx.cancel_all() → Agent 停止
```

#### 修复过程中的关键 Bug
- **reader.cancel() 不关闭 TCP 连接**: 浏览器 cancel ReadableStream 不一定关连接，Java 写 TCP buffer 照常成功。解决：增加显式 `DELETE /cancel` HTTP 请求。
- **cancel 帧被中断**: `send().subscribe()` 后立即 `disposable.dispose()` 取消正在发送的帧。解决：send complete/error 回调中才调 `cancelExistingSession`。
- **重复清理**: `agent_finish` 时 `cleanup()` + `emitter.complete()` → `onCompletion` → `cancelSession` 双重清理。解决：流模式移除非 emitter 回调的 `cleanup`。

### 一、Chat 组件 LaTeX 公式渲染修复

#### 问题描述
AI 助手返回的数学公式使用 `[...]` 包裹（如 `[\frac{a}{b}]`），但 markstream-vue 只识别标准的 `$...$` 和 `$$...$$` LaTeX 格式，导致数学公式无法渲染显示。

#### 方案选型过程
**方案一：前端正则预处理**
- 在 `index.vue` 的 `onMessage` 回调中，对 AI 返回数据做预处理——将 `[...]` 格式转为 `$$...$$` 格式
- 核心思路：多层安全过滤确保不误判
  - 第1层：必须独占一行，排除 `[1,2,3]`、`[NOTE]`、`[link](url)` 等内联用法
  - 第2层：必须包含 LaTeX 特征（如 `\frac`、`\sum`、`^{`、`_{` 等）
  - 第3层：长度阈值 + 排除列表
- 缺点：改动量大（~50行正则）、误判风险需持续维护排除列表

**方案二：AI Engine System Prompt（最终选择）**
从源头解决，让 LLM 直接输出标准格式。

#### 最终代码改动
修改文件：`jzo2o-ai-engine/app/api/chat.py`

```python
SYSTEM_PROMPT = """你是一个AI助手。请严格遵循以下输出格式要求：

1. 数学公式必须使用标准 Markdown LaTeX 格式：
   - 行内公式用单个 $ 包裹，例如 $E=mc^2$
   - 块级/独立公式用双 $$ 包裹，例如 $$\\frac{a}{b}$$
   - 禁止使用 [ ... ] 或 \\[ ... \\] 作为公式分隔符

2. 支持渲染 Mermaid 图表，使用 ```mermaid 代码块

请确保所有数学表达式都使用 $ 或 $$ 包裹。"""

# 在请求处理时注入
messages_dict = [m.model_dump() for m in request.messages]
messages_dict.insert(0, {"role": "system", "content": SYSTEM_PROMPT})
```

#### 方案对比
| 方案 | 改动量 | 误判风险 | 维护成本 | 推荐度 |
|------|--------|----------|----------|--------|
| 前端正则预处理 | ~50行 | 有（多层防护复杂） | 高 | ❌ |
| 后端 System Prompt | 5行 | 无 | 零 | ✅ |

#### 效果
从源头强制 LLM 输出标准格式，彻底解决渲染问题，无需前端做任何特殊处理。重启 AI Engine 服务即可生效。

### 三、悬浮窗切换侧边栏时不自动关闭的 Bug 修复

#### 问题描述
在其他界面点击悬浮球打开悬浮窗后，切换到侧边栏的 AI 聊天界面时，悬浮窗不会自动关闭。

#### 根因分析
在 `layouts/index.vue` 中：
- `showChat` 是本地 `ref(false)` 状态
- `<ChatFloatingButton>` 有 `v-if="chatStore.mode !== 'sidebar'"` 条件，切换到侧边栏时会隐藏
- 但 `<ChatWindow :visible="showChat">` 没有条件判断，且 `showChat` 仍为 `true`

#### 解决方案
添加 `watch` 监听 `chatStore.mode`，当变为 `'sidebar'` 时自动关闭悬浮窗。

#### 代码改动
修改文件：`project-xzb-PC-vue3-java/src/layouts/index.vue`

```typescript
// 当进入侧边栏聊天页时，自动关闭悬浮窗
watch(
  () => chatStore.mode,
  (newMode) => {
    if (newMode === 'sidebar') {
      showChat.value = false
    }
  }
)
```

### 四、SQLite Checkpoint 数据库路径分析

#### 问题描述
确认 AI Engine 中用于维持对话状态的 SQLite 数据库在代码中的位置是使用相对路径还是绝对路径。

#### 代码分析

**1. `.env` 文件**
```
CHECKPOINT_DB_PATH=data/checkpoints.db
```

**2. `config.py`（第24-25行）**
```python
checkpoint_db_path: str = os.getenv("CHECKPOINT_DB_PATH", "data/checkpoints.db")
```

**3. 实际使用位置 — `graph.py`**
```python
db_path = settings.checkpoint_db_path    # ← 拿到的是 "data/checkpoints.db"
if db_path:
    Path(db_path).parent.mkdir(parents=True, exist_ok=True)
    conn = await aiosqlite.connect(db_path)
```

#### 结论
| 项目 | 说明 |
|------|------|
| **路径类型** | **相对路径** (`data/checkpoints.db`) |

### 六、聊天界面自动滚动问题分析

#### 问题描述
当前无论是悬浮窗还是侧边栏的聊天界面，AI 生成信息时界面不能自动滚动到最新内容位置。

#### TDesign ChatList 原生支持

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `autoScroll` | Boolean | **true** | 是否开启自动滚动 |
| `showScrollButton` | Boolean | true | 是否显示回到底部按钮 |

#### 问题根因：**双重滚动容器冲突**

代码结构存在嵌套滚动：
```vue
<!-- ChatMessageList.vue -->
<div class="chat-message-list" style="overflow-y: auto;">  <!-- ❌ 外层滚动 -->
  <ChatList :auto-scroll="true">                             <!-- ⚠️ 内层也有滚动 -->
    ...
  </ChatList>
</div>
```

TDesign ChatList 内部的 `.chat__list` 本身就是滚动容器，外层再包一个 `overflow-y: auto` 导致：
1. `ResizeObserver` 检测到的尺寸变化无法正确映射到实际滚动位置
2. 滚动事件被外层拦截，自动滚动行为异常

#### 解决方案
移除外层的 `overflow-y: auto`，让 TDesign ChatList 全权管理滚动。
