# 开发日志 — 2026-05-25

## 一、AI 聊天记忆机制全链路分析

### 1.1 记忆来源

AI 的"记忆"**不是 Python 引擎自己查数据库维护的**，而是由前端/Java 端预先组装好完整消息列表后传入。

```
前端 (TDesign Chatbot 组件内内存)
  │  取最近 20 条 (slice(-20))
  │  + 追加当前用户输入
  ▼
Java (ChatServiceImpl)
  │  convertMessages() 原样透传
  ▼
Python (ws_chat.py)
  │  _dicts_to_messages()
  │  传给 LangChain Agent
  ▼
LLM (DeepSeek 等)
```

### 1.2 两层"状态保持"机制

| 层次 | 机制 | 存储位置 | 定位 |
|------|------|---------|------|
| **业务层** | 前端组装完整消息数组传给后端 | 前端内存 / DB | 主要记忆来源 |
| **框架层** | LangGraph AsyncSqliteSaver 持久化 checkpoint | `data/checkpoints.db` | 容错/恢复辅助 |

### 1.3 记忆生命周期

- **同一会话内连续对话**：前端 TDesign Chatbot 组件内部状态（`messagesStore`）保存最近消息，每次请求取 `slice(-20)` 发给后端，**不查数据库**
- **切换到历史会话**：前端调 `getSessionMessages(sessionId)` → Java 从 `ai_chat_record` 表查全部历史 → 写入 Chatbot 组件 → 之后走正常内存路径
- **持久化写入**：每次发消息时 Java 端同步 `saveRecord(user)`，AI 回复完成后回调 `saveRecord(assistant)`

## 二、通信模式分析

### 2.1 Java ↔ Python 两种通信方式

通过 `jzo2.ai.engine.mode` 配置切换：

| 维度 | WebSocket 模式 (`mode=ws`) | HTTP 模式 (`mode=http`) |
|------|---------------------------|------------------------|
| 端点 | `/ws/chat/{session_id}` | `/chat/completions` |
| 协议 | WebSocket 全双工 | HTTP POST + SSE 单向流 |
| 工具调用 | ✅ 支持 (tool_call/tool_result 往返) | ❌ 不支持 |
| 会话状态 | per-session 长连接 + 空闲超时释放 | 无状态 |
| 当前默认 | ✅ 默认使用 | 保留兼容 |

### 2.2 评论总结功能实际用 WS 的方式

`EvaluationSummaryServiceImpl.summarize()` 用了 `wsClient.connectAndCollect()` 模式，但实际**只用 WS 发了一条单轮 user_message，等 LLM 返回完整文本**。数据准备全在 Java 端完成（DB 查旧总结 + Feign 查新评价 + 拼 prompt），WS 的 tool_call/tool_result 等多轮双向能力完全没有用到。这个场景用 HTTP 即可，属于用 WS 杀鸡用了牛刀。

### 2.3 WS 工具调用 vs 两次 HTTP 对比

| 维度 | WS 方案 | 两次 HTTP |
|------|---------|----------|
| 会话绑定 | 天然通过 WS session 绑定 userId/userType | 需额外传认证信息 |
| 用户身份传递 | Java SessionContext 直接持有 | Python 需传 Header |
| 多轮工具调用 | ✅ 同一 WS 上反复 tool_call/tool_result | ⚠️ 每次独立 HTTP 请求 |
| 取消能力 | ✅ cancel 帧立即中断 | ❌ 已发 HTTP 无法取消 |
| 进度通知 | ✅ tool_start/tool_end 推前端 | ❌ 前端不知道后台在做什么 |

## 三、评价总结功能设计评审

### 3.1 当前实现结论

`summarize()` 方法是纯粹的单轮请求-响应模型：
```
Java 拼好所有数据 → 调 LLM 生成一段文本 → 存回数据库
```
不需要 WS 的有状态、双向、多轮能力。

### 3.2 双路径架构设计（聊天触发 + 按钮触发）

```
┌─────────────────────────────────────┐
│      EvaluationSummaryCore (共享层)  │
│  · prepareData()  — 数据准备        │
│  · buildPrompt()  — Prompt 模板     │
│  · saveSummary()  — 结果持久化      │
├─────────────────┬───────────────────┤
│  路径 A: Agent  │  路径 B: Direct   │
│  (聊天对话触发)  │  (按钮/定时触发)  │
│  LLM 自主决策   │  Java 预先取数据  │
│  WS 工具调用    │  HTTP 单轮调用    │
└─────────────────┴───────────────────┘
```

两个路径的差异：
| | 路径 A (聊天) | 路径 B (按钮) |
|--|-------------|--------------|
| 谁决定调什么 | LLM 自主决定 | Java 代码写死 |
| 数据获取方式 | LLM 通过 WS 工具逐个查 | Java 一次全准备好 |
| 通信方式 | WS (tool_call/tool_result) | HTTP (POST /chat/completions) |

## 四、聊天消息落盘机制分析

### 4.1 当前落盘链路

`ChatServiceImpl.chat()` 中有两个落盘点：

| | 落盘点 ① (user 消息) | 落盘点 ② (assistant 消息) |
|--|----------------------|--------------------------|
| 触发时机 | AI 调用**之前**，同步执行 | AI 回复**完成之后**，`onCompleteCallback` 回调触发 |
| 可靠性 | **100%** | **不可靠** |

### 4.2 当前方案的问题：7 条链路中 6 条丢失 assistant 消息

| # | 场景 | assistant 落盘 |
|---|------|----------------|
| N1 | Agent 正常完成 (`agent_finish` 帧) | ✅ `onCompleteCallback` |
| E1 | Python 返回 error 帧 | ❌ 未调用 |
| E2 | WebSocket 连接断开 | ❌ 未调用 |
| E3 | 用户主动取消 | ❌ 未调用 |
| E4 | 用户刷新/关闭页面 | ❌ 未调用 |
| E5 | SSE 超时 (5min) | ❌ 未调用 |
| E6 | emitter.send 抛 IO 异常 | ❌ 未调用 |

## 五、增量异步持久化方案设计

### 5.1 方案演进过程

| 版本 | 策略 | 用户反馈 |
|------|------|---------|
| v1 | 每个 token 都落盘 | "不能立即落盘" |
| v2 | 每 N 个 token 落一次 | "定时落盘即可" |
| v3 | 每 500ms 定时落盘 | "落盘要异步落盘" |
| v4 (最终) | **异步 500ms 定时落盘 + 三回调覆盖** | ✅ 确认 |

### 5.2 数据模型变更

`ai_chat_record` 表新增 `status` 字段（4 态状态机）：

```sql
ALTER TABLE ai_chat_record ADD COLUMN status TINYINT NOT NULL DEFAULT 1
  COMMENT '消息状态: 0-生成中, 1-正常完成, 2-用户取消, 3-异常中断';
```

| status | 含义 | content 处理 | 前端展示 |
|--------|------|-------------|---------|
| 0 | 生成中 | 当前已有内容 | 加载动画 |
| 1 | 正常完成 | 完整回复 | 正常展示 |
| 2 | 用户取消 | 已有部分内容，**不拼接后缀** | 可选灰字"已停止生成" |
| 3 | 异常中断 | 已有内容 + `\n\n回复未完成，输入继续以继续` | 展示 + 提示文字 |

### 5.3 核心设计：用户取消 vs 异常中断的区分

| 维度 | status=2 用户取消 | status=3 异常中断 |
|------|-------------------|-------------------|
| 触发源 | 用户点击停止 / 刷新页面 / 关标签 | Python error 帧 / WS 断连 / SSE 超时 |
| 语义 | 用户主动行为，预期内 | 系统故障，非预期 |
| 上下文污染 | 不污染后续对话 | 需告知用户并允许恢复 |
| 提示文案 | 不拼接（仅前端轻量提示） | 拼接 "回复未完成，输入继续以继续" |
| 数据保留 | 已收到的 token 仍需落盘 | 同样落盘 |

### 5.4 回调分发矩阵

`cleanup()` 方法根据 `cancelReason` 分发到不同回调：

| 触发场景 | cancelReason | 调用的回调 | 最终 status |
|---------|-------------|-----------|------------|
| 收到 `agent_finish` 帧 | 不走 cleanup | `onCompleteCallback` | 1 |
| `ChatServiceImpl.cancel()` | `"user_cancel"` | `onCancelCallback` | 2 |
| `emitter.onCompletion()` | `"user_cancel"` | `onCancelCallback` | 2 |
| `emitter.onTimeout()` | `"timeout"` | `onCancelCallback` | 2 |
| `emitter.onError()` | `"sse_error"` | `onErrorCallback` | 3 |
| 收到 `error` 帧 | `"python_error"` | `onErrorCallback` | 3 |
| WS 连接异常断开 | `"ws_error"` | `onErrorCallback` | 3 |
| WS 消息 IO 异常 | `"ws_error"` | `onErrorCallback` | 3 |

### 5.5 AsyncFlusher 异步落盘器设计

**职责**：预 INSERT 空壳记录 → 每 500ms 增量 UPDATE → 结束时 finalize

**生命周期**：
1. `chat()` 开始：立即 INSERT status=0 记录，启动 500ms 定时 flush
2. `token` 回调：不直接写 DB，仅追加 buffer
3. 定时任务：每 500ms `UPDATE content WHERE id=? AND status=0`
4. 结束：`finalize(status, suffix)` — 停止定时器 → 写入最终状态
5. `finalize()` 幂等（`AtomicBoolean` guard）

**边界处理**：
- 空记录清理：用户取消时若 buffer 为空 → DELETE 空记录
- finalize 并发竞争：`AtomicBoolean.compareAndSet` 保证只执行一次
- flush 与 finalize 并发：flush 检查 `finalized` 标记；DB 层 UPDATE 带 `WHERE status=0`
- 线程池满：有界队列 + CallerRunsPolicy，调用线程兜底执行

### 5.6 线程模型

```
Tomcat HTTP Thread
  ├─ sync: saveRecord(user)
  ├─ new AsyncFlusher()     ← 预 INSERT
  ├─ wsClient.connectAndStream()
  └─ return SseEmitter

async-flush-pool (coreSize=2, 共享)
  └─ AsyncFlusher#flush (每 500ms)

reactor-netty EventLoop
  ├─ WS 连接 I/O
  ├─ token → emitter.send()
  └─ token → accumulator.accept()
```

## 六、关键经验总结

| 发现 | 说明 |
|------|------|
| 前端 `slice(-20)` 截断过于粗暴 | 固定条数 ≠ 固定 token 数，长消息可能超上下文，短消息浪费窗口 |
| 前端承担了不该承担的职责 | 记忆管理、截断逻辑、消息格式转换全在前端，且两处重复实现 |
| 缺乏摘要/压缩机制 | 超过 20 条直接丢弃早期信息，没有"沉淀"用户画像 |
| 评论总结功能用 WS 属于过度设计 | 本质是单轮请求-响应，HTTP 即可 |
| assistant 落盘仅覆盖 1/7 路径 | 剩余 6 条异常路径全部丢失数据 |
| 用户取消 ≠ 系统异常 | 应区分对待，语义完全不同 |
